# Story 6.3 — Validacao de Campos Obrigatorios por Status

## Status
Done

## Executor
@dev (Dex)

## PRD
PRD-001 — Apolizza CRM (Fase 1: CRUD + Migracao)

## Story
Como **gestor da corretora**, quero que o sistema bloqueie a mudanca de status se os campos obrigatorios nao estiverem preenchidos, para garantir a qualidade dos dados em cada etapa do processo.

## Acceptance Criteria

- [x] AC1: Ao mudar status para "nao iniciado", sistema exige 7 campos (testado)
- [x] AC2: Ao mudar status para "fechado", sistema exige 5 campos (testado via curl)
- [x] AC3: Ao mudar status para "perda", sistema exige Valor em Perda (testado via curl)
- [x] AC4: Validacao dual: frontend (CotacaoForm) + backend (API PUT)
- [x] AC5: Campos faltantes listados em mensagem de erro clara
- [x] AC6: Campos faltantes destacados com borda vermelha + bg-red-50
- [x] AC7: Regras vem da tabela `status_config.required_fields` via `/api/status-config`
- [x] AC8: Mudanca permitida quando todos campos preenchidos (testado)
- [x] AC9: Status sem regras permitem mudanca livre (testado)

## Tasks

- [x] Criar funcao `validateStatusFields()` compartilhada (backend + frontend)
- [x] Integrar validacao no endpoint `PUT /api/cotacoes/[id]` quando status muda
- [x] Integrar validacao no frontend CotacaoForm pre-submit
- [x] Buscar regras via `GET /api/status-config` no frontend
- [x] Destaque visual: borda vermelha + bg-red-50 nos campos faltantes
- [x] Mensagem de erro listando campos faltantes por nome
- [x] Testado: fechado sem campos → 422, fechado com campos → 200
- [x] Testado: perda sem valor_perda → 422, em andamento livre → 200

## Dev Notes

- **Regras na tabela:** `status_config.required_fields` e um JSONB array: `["fim_vigencia", "inicio_vigencia", ...]`
- **Dual validation:** Frontend valida para UX rapido, backend valida para seguranca (nunca confiar so no client)
- **Cache:** As regras de validacao raramente mudam — cachear no frontend e no backend
- **Reaproveitamento:** A logica de `checkRequiredFields` do dashboard.html v1 e a mesma — agora via SQL
- **Depende de:** Story 5.1 (tabela status_config com seed), Story 6.1/6.2 (formularios)
